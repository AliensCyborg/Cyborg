Is workflow ka kaam: **sirf planning generate** karna for a single PyOS Python Engine.

Example:
- `Workflow("PyOS_Engine_Python", "PyOS.Cache", "Enterprise caching engine..." )`
- `Workflow("PyOS_Engine_Python", "PyOS.Troubleshoot", "New engine (name only). Requirements optional." )`

Output:
- `/Aliens/.Alien/{Developer}/Planning/PyOS/Engine/py/{EngineBaseName}.plan.md`

## [02] Target Normalization (PyOS Engine)
Given `Targets`:
- Trim whitespace
- If target does not start with `PyOS.` then prefix `PyOS.`
- Validate EngineShortName:
  - Must match: `^[A-Za-z][A-Za-z0-9]*$`
  - Examples: `Cache`, `API`, `ABTest`

Canonical Engine Base Name:
- `EngineBaseName = "PyOS.{EngineShortName}"`

## [04] Evidence Pack (Minimum)
Planning must record evidence sources (best-effort):
1) EngineRouter rules:
   - `/Aliens/PyOS/{Version}/Boot/PyOS.EngineRouter.py`
2) Existing engine file (if present):
   - `/Aliens/PyOS/{Version}/Engine/{EngineBaseName}.py`
3) At least 2–4 example engines (read-only) to match style.

If evidence is missing:
- Do not guess hidden facts.
- Planning must explicitly mark fields as "Not found".

---

## [05] Engine Type Classification (Critical)
Because PyOS static facade routes through EngineRouter, default engine type must be:
- `EngineRouterStatic` (recommended & generated by Code workflow)
  - `class PyOS_{EngineShortName}`
  - `public static function Main(array $Payload, array $Context)`
  - Actions (if any) are also `public static` and accept `(array $Payload, array $Context)`.

If an existing engine is detected as instance-based / non-router compliant (e.g., no static Main):
- Mark `LegacyNonCompliant = true`
- Planning must propose 2 safe options:
  1) **Do not change** engine behavior (document-only)
  2) Create a **router-compatible facade** (high risk) → requires manual approval

Planning must NOT auto-approve conversions.

9) Outputs (exact paths)

---

## [06.1] Enterprise Planning Expectations (Ultra)
Planning MUST additionally include these enterprise sections (append after the fixed template, do not replace it):

12) Coverage Inventory (No Missing Coverage)
- Repo discovery se list banao:
  - `PyOS::{EngineShortName}{Action}(` patterns (best-effort)
  - existing engine file me already defined public static methods
- Output table/list:
  - ActionName
  - ExistsAlready (yes/no)
  - Plan: implement | stub-safe | document-only
  - Risk note (if any)

13) Enterprise Baseline Features
- Standard helpers plan karo (engine ke andar) so future calls break na hon:
  - payload arg extraction helper (EngineRouter numeric payload)
  - validation helpers (required fields, types, path/host/id/email/name etc as relevant)
  - error model (codes + messages)
  - redact/sanitize for logs (no secrets)

14) Large-Engine Policy
- Agar engine foundational/cross-cutting ho (Security/Auth/API/Cache):
  - plan MUST be "comprehensive implementation" not "minimal skeleton".
  - explicitly note: "large file is expected".

15) Return-Type Expectations (Compatibility)
- Repo callsites sometimes treat the same engine as:
  - `bool` (if used in `if (!PyOS::XyzAction(...))`)
  - `array` envelope (expects keys like `success`, `data`)
  - `string` (used as token/url/hash/random)
- Planning MUST include a small matrix:
  - ActionName → ExpectedReturnType(s) → Evidence (file + pattern)
- If ambiguous:
  - prefer backward compatible behavior,
  - document explicitly (do not guess),
  - add safe fallback return that does not fatal.

16) EngineRouter Payload Passing (Numeric Args)
- Planning MUST call out that EngineRouter passes call args as numeric `$Payload`:
  - `PyOS::EngineAction($a,$b)` → `$Payload = [$a,$b]`
- Plan MUST include "payload arg extraction" helper requirement for foundational engines.

17) NO Jugaad Policy (Hard)
- Planning MUST explicitly state: "Main-only / skeleton engine is forbidden".
- Planning MUST list the **minimum baseline** that MUST exist in code output:
  - CodeHeader (with Dependencies)
  - CodeFooter
  - guarded logger helper
  - payload arg extraction helper
  - structured Ok/Fail/error model helpers

18) Enterprise Feature Blueprint (Topic-based)
- Planning MUST define the engine's topic feature-set (large + exhaustive allowed/expected).
- This blueprint is not a single-line bullet; it must be a structured list of capabilities, grouped by:
  - Core features
  - Security/validation
  - Logging/audit
  - Performance/caching (if relevant)
  - Developer UX (easy short powerful usage via facade)

19) Integration Map (Must)
- Planning MUST list which existing PyOS engines will be used and why (avoid reinventing systems).
- Example categories:
  - `PyOS::Security*()` for validation
  - `PyOS::Policy*()` for permissions
  - `PyOS::Log()` for audit logs
  - `PyOS::Cache()` for performance
